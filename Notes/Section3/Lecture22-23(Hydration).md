In the previous lecture, we learned:

```text

```

and

```text
Client Components

↓
```

> If HTML is already visible in the browser, why does React execute the component again?

The answer is:

# 🚀 Hydration

Hydration is one of the most important concepts in React and Next.js.

Without understanding Hydration, concepts like:

- Client Components
- useState
- useEffect
- onClick
- Hydration Errors

become difficult to understand.

---

# What is Hydration?

Hydration is the process where React attaches JavaScript functionality to HTML received from the server.

Think of it like:

```text
Server

↓

Send HTML

↓

Browser Displays HTML

↓

React Attaches Events

↓

Page Becomes Interactive
```

---

# Why Do We Need Hydration?

Suppose server sends:

```html
<button>Like</button>
```

Browser displays:

```text
Like
```

But clicking it does nothing.

Why?

Because HTML alone doesn't contain React functionality.

---

To make it interactive:

```text
React Loads JavaScript

↓

Finds Existing HTML

↓

Connects Event Handlers

↓

Button Starts Working
```

This process is called:

```text
Hydration
```

---

# Real-Life Example

Imagine buying a mobile phone.

Initially:

```text
Phone Body
```

exists.

But:

```text
No Battery
```

No power.

No functionality.

---

After inserting battery:

```text
Phone Works
```

Hydration is similar.

Server sends:

```text
HTML Body
```

Hydration adds:

```text
JavaScript Power
```

making the page interactive.

---

# When Does Hydration Run?

Hydration runs whenever a Client Component is rendered in the browser.

---

## Initial Page Load

Example:

```text
Open Website
```

Hydration runs.

---

## Refresh

Example:

```text
F5
```

Hydration runs again.

---

## New Page Request

Example:

```text
Open New Tab
```

Hydration runs.

---

# Important Note

Hydration is mainly associated with:

```text
Client Components
```

because they contain:

```text
Events

Hooks

Browser APIs
```

that need JavaScript.

Server Components themselves do not become interactive.

---

# What is a Hydration Error?

Hydration Error occurs when:

```text
Server HTML

≠

Client HTML
```

during hydration.

---

React expects:

```text
Same HTML
```

on both sides.

If they differ:

```text
Hydration Error
```

appears.

---

# Easy Definition

Hydration Error means:

> The HTML generated on the server is different from the HTML generated by React in the browser during hydration.

---

# Your Example

```jsx
export default function Likes() {
  if (typeof window === "undefined") {
    return <div>sdbfekjb,dskvdsbsbduvdskscjdshvdc</div>;
  }

  return <div>Likes 2k</div>;
}
```

---

# What Happens?

## Server

Server doesn't have:

```text
window
```

Therefore:

```jsx
typeof window === "undefined";
```

becomes:

```text
true
```

Server renders:

```html
<div>sdbfekjb,dskvdsbsbduvdskscjdshvdc</div>
```

---

## Browser

Browser has:

```text
window
```

Therefore:

```jsx
typeof window === "undefined";
```

becomes:

```text
false
```

Browser renders:

```html
<div>Likes 2k</div>
```

---

Comparison:

```text
Server

↓

sdbfekjb,dskvdsbsbduvdskscjdshvdc

--------------------

Browser

↓

Likes 2k
```

Different outputs.

React says:

```text
Something Is Wrong
```

Hydration Error.

---

# Why Does This Cause an Error?

React expects:

```text
Server HTML

↓

Exactly Match

↓

Client HTML
```

because hydration attaches behavior to existing HTML.

Different HTML means:

```text
React Doesn't Know
Which DOM To Trust
```

---

# Correct Version

```jsx
export default function Likes() {
  if (typeof window === "undefined") {
    return <div>Likes 2k</div>;
  }

  return <div>Likes 2k</div>;
}
```

Server:

```text
Likes 2k
```

Browser:

```text
Likes 2k
```

Same output.

No hydration error.

---

# Does Hydration Error Only Happen in Client Components?

Mostly:

```text
Yes
```

because hydration itself happens for Client Components.

Server Components don't hydrate in the browser.

---

More accurate statement:

```text
Hydration Errors are usually caused by Client Components
because they are rendered both on the server and browser.
```

---

# Why Does Next.js Show It Mostly in Development?

Development Mode:

```text
Strict Validation
```

React checks:

```text
Server Output

vs

Client Output
```

and warns immediately.

---

Production Mode:

React tries to recover automatically whenever possible.

Therefore warnings may disappear.

---

# Important Question

If Production Doesn't Show Errors,

Why Fix Them?

Because:

```text
No Warning

≠

No Problem
```

# Real Example

Server:

```html
<p>Price ₹100</p>
```

Browser:

```html
<p>Price ₹200</p>
```

Hydration mismatch.

Production may silently replace:

```text
₹100

↓

₹200
```

User briefly sees:

```text
Wrong Data
```

This causes:

- UI Flickering
- Layout Shifts
- Inconsistent State
- Unexpected Bugs

Therefore hydration errors should always be fixed.

# Common Causes of Hydration Errors

## 1. Using Date()

```jsx
<div>{new Date()}</div>
```

---

Server:

```text
10:00:00
```

Browser:

```text
10:00:01
```

Different output.

Hydration error.

---

## 2. Using Math.random()

```jsx
<div>{Math.random()}</div>
```

Server:

```text
0.53
```

Browser:

```text
0.91
```

Mismatch.

---

## 3. Using Window During Render

```jsx
<div>{window.innerWidth}</div>
```

Server:

```text
window undefined
```

Browser:

```text
1200
```

Mismatch.

---

## 4. Using LocalStorage During Render

```jsx
<div>{localStorage.getItem("theme")}</div>
```

Server:

```text
No localStorage
```

Browser:

```text
dark
```

Mismatch.

---

## 5. Conditional Rendering Based On Browser APIs

```jsx
if (typeof window !== "undefined") {
  return <div>Desktop</div>;
}

return <div>Server</div>;
```

Different output.

Hydration error.

---

# How To Avoid Hydration Errors?

## Method 1

Use:

```jsx
useEffect();
```

for browser-only code.

Example:

```jsx
"use client";

import { useEffect } from "react";

export default function Page() {
  useEffect(() => {
    console.log(window.innerWidth);
  }, []);

  return <h1>Hello</h1>;
}
```

---

Why?

Because:

```text
useEffect

↓

Runs After Hydration

↓

Browser Only
```

---

## Method 2

Keep Initial UI Same

Server:

```text
Loading...
```

Browser:

```text
Loading...
```

Hydration succeeds.

Later update state.

---

# Example

```jsx
"use client";

import { useState, useEffect } from "react";

export default function Width() {
  const [width, setWidth] = useState(null);

  useEffect(() => {
    setWidth(window.innerWidth);
  }, []);

  return <div>Width: {width}</div>;
}
```

No hydration mismatch.

# Key Takeaways

- Hydration attaches JavaScript functionality to server-rendered HTML.
- Hydration mainly affects Client Components.
- Hydration Errors occur when server HTML and client HTML differ.
- Production may hide warnings but the bug still exists.
- Browser-only APIs are a common source of hydration errors.
- useEffect is a common solution for browser-only logic.

# Interview Questions

### Q1. What is Hydration in React/Next.js?

Hydration is the process where React attaches event handlers and interactivity to HTML received from the server.

---

### Q2. What is a Hydration Error?

A Hydration Error occurs when the HTML generated on the server differs from the HTML generated in the browser during hydration.

---

### Q3. Why are hydration errors dangerous?

They can cause UI inconsistencies, flickering, incorrect state, and unpredictable behavior.

---

### Q4. How can useEffect help prevent hydration errors?

Because useEffect runs only in the browser after hydration, allowing browser-specific logic to execute safely.

---

### Q5. Why do Client Components experience hydration?

Because they must become interactive in the browser by attaching JavaScript behavior to existing HTML.

# Summary

In these lectures, we learned that Hydration is the process of attaching JavaScript functionality to server-rendered HTML, making pages interactive. We also learned that Hydration Errors occur when the server-rendered HTML differs from the client-rendered HTML during hydration. Common causes include using browser-only APIs, random values, dates, and conditional rendering based on the environment. Although production mode may hide warnings, hydration mismatches can still lead to subtle bugs, so they should always be fixed.
